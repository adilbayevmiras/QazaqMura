# QAZAQ MŪRA — Development Guide
### Руководство по разработке для Cursor · v1.0 · для соло-фаундера, работающего с AI

*Собрано на основе: Продуктовой спецификации v1.0, Wireframe Architecture, Визуальной дизайн-системы v1.0. Это не код — это точная последовательность, которой Cursor должен следовать, фаза за фазой, не забегая вперёд.*

---

## 0. Как пользоваться этим гайдом

1. **Одна фаза = одна рабочая сессия с Cursor = один PR/коммит в main.** Не начинай следующую фазу, пока не выполнены Exit Criteria текущей — Cursor склонен "забегать вперёд" (начинать бронирование, пока ещё нет контента), это и есть главный риск соло-разработки с AI без плана.
2. **В начале каждой сессии** прикрепляй к контексту Cursor: этот гайд (текущую фазу), Продуктовую спецификацию (соответствующий раздел), Wireframe Architecture (соответствующие экраны), Design System (соответствующие компоненты). Без этого Cursor будет придумывать структуру данных и стили заново на каждой фазе.
3. **После каждой фазы — прогоняй Testing Checklist руками**, не доверяй "судя по коду должно работать". Соло-фаундер без QA-команды — единственный, кто поймает регресс до продакшена.
4. **Коммить мелко, внутри фазы.** Каждая задача (Tasks) — отдельный коммит с понятным сообщением, чтобы можно было откатить один шаг, не откатывая всю фазу.
5. **Фазы 0–5 — обязательно последовательны.** Фазы 9–15 (Marketplace, Payments, Community и т.д.) можно частично переставлять местами под приоритеты бизнеса, но зависимости (указаны в каждой фазе) — жёсткие, не нарушать.

### 0.1 Зафиксированные решения по репозиторию (не открыто к обсуждению на старте)
- **Монорепозиторий**: pnpm workspaces + Turborepo. *Почему*: три приложения (web, api, cms) с общими типами/токенами дизайн-системы — раздельные репозитории означают дублирование и рассинхрон типов на старте, когда это критичнее всего избежать.
- Структура верхнего уровня:
```
/apps
  /web        — Next.js (публичный сайт + личный кабинет)
  /api        — NestJS (Content/Search/Booking/Payments/Payouts/Reviews/Notifications)
  /cms        — Strapi (редакторская админка контента)
/packages
  /ui         — общая библиотека компонентов дизайн-системы
  /config     — общие ESLint/TS/Tailwind конфиги
  /types      — общие TypeScript-типы data model (Tradition, Organization, Master, Experience…)
```

## Фаза 0 — Инфраструктурный фундамент

**Goal**: рабочее окружение, в котором любая следующая фаза воспроизводима одной командой, до того как написана хоть одна продуктовая строка кода.

**Tasks**:
- Инициализировать monorepo (pnpm + Turborepo), базовые `apps/web`, `apps/api`, `apps/cms`, `packages/ui`, `packages/config`, `packages/types`.
- Настроить единый ESLint/Prettier/TypeScript strict-конфиг во всех пакетах.
- Поднять локальный docker-compose: PostgreSQL+PostGIS, Redis, Meilisearch.
- Настроить переменные окружения (`.env.example` для каждого приложения) и секреты (не коммитить).
- Настроить CI (GitHub Actions): lint + typecheck + build на каждый PR.
- Завести Git-репозиторий, ветки `main`/`dev`, правило "не пушить напрямую в main".

**Files**:
```
/turbo.json, /pnpm-workspace.yaml
/apps/web/.env.example, /apps/api/.env.example, /apps/cms/.env.example
/docker-compose.yml
/.github/workflows/ci.yml
/packages/config/eslint-preset.js, /packages/config/tsconfig.base.json
```

**Technologies**: pnpm, Turborepo, Docker Compose, GitHub Actions, TypeScript.

**Expected result**: `pnpm install && docker compose up && pnpm dev` поднимает все три приложения локально без ошибок; пустой CI зелёный.

**Testing checklist**:
- [ ] Чистый клон репозитория на другой машине/директории поднимается по инструкции из README без ручных правок
- [ ] `pnpm lint` и `pnpm typecheck` проходят на пустых заглушках
- [ ] docker-compose поднимает PostgreSQL с включённым расширением PostGIS (проверить `SELECT PostGIS_version();`)
- [ ] CI-пайплайн падает при намеренно сломанном коде (проверка, что CI реально проверяет, а не всегда зелёный)

**Exit criteria**: три пустых приложения запускаются локально, CI работает, окружение задокументировано в README так, что сам фаундер сможет восстановить его через полгода без потери контекста.

## Фаза 1 — Дизайн-система в коде

**Goal**: библиотека UI-компонентов, реализующая Design System v1.0 (§3–§8 дизайн-системы), до начала любых продуктовых экранов — иначе каждая следующая фаза будет заново придумывать кнопки и отступы.

**Tasks**:
- Настроить Tailwind (или CSS-in-JS — зафиксировать один подход) с токенами из дизайн-системы: цвета (§3), типографика (§4), spacing (§6), radius (§6), breakpoints (§7).
- Реализовать базовые примитивы: Button (варианты §8.2), Input/Form-controls (§8.3), Card (варианты §8.1), Pill/Chip, Icon-обёртку над line-библиотекой.
- Реализовать светлую/тёмную тему как переключаемый контекст (§13 дизайн-системы) — тема должна переключаться уже на этой фазе, на пустых компонентах.
- Собрать Storybook (или аналог) для визуальной проверки компонентов в изоляции.

**Files**:
```
/packages/ui/src/tokens/{colors,typography,spacing,radius}.ts
/packages/ui/src/components/{Button,Input,Card,Pill,Icon}/
/packages/ui/src/theme/ThemeProvider.tsx
/packages/ui/.storybook/
```

**Technologies**: Tailwind CSS (с кастомной темой из токенов), React, Storybook.

**Expected result**: Storybook с рабочими Button/Input/Card в двух темах, визуально соответствующими Design System v1.0 (не "похоже", а по конкретным hex/spacing-значениям из документа).

**Testing checklist**:
- [ ] Каждый компонент проверен в light и dark теме визуально рядом (side-by-side в Storybook)
- [ ] Фокус клавиатурой виден на всех интерактивных компонентах (§15 дизайн-системы)
- [ ] Отступы/радиусы сверены с таблицами §6 дизайн-системы вручную хотя бы на 3 компонентах
- [ ] Контраст текста в обеих темах проверен инструментом (напр. встроенный Storybook a11y addon)

**Exit criteria**: библиотека компонентов опубликована как внутренний workspace-пакет, импортируется в `apps/web` без ошибок сборки, переключение темы работает end-to-end.

## Фаза 2 — Контент-модель и CMS

**Goal**: редакторская система для контента традиций, готовая для заполнения реальными данными (после сверки с gov.kz/UNESCO/nmrk.kz — Фаза 0 бизнес-roadmap'а спецификации, юридически/фактически должна пройти до этой инженерной фазы).

**Tasks**:
- Развернуть Strapi в `apps/cms`, настроить типы контента по data model §5.2 спецификации: `Tradition`, `TraditionTranslation` (kk/ru/en, `translation_status`), `Category`, `Region`, `City` (геоиерархия §4 спецификации).
- Настроить обязательные поля: `heritage_status`, `source_refs[]` (source/url/retrieved_at), `status` (draft/needs_verification/published), `last_verified_at`.
- Настроить медиа-поля галереи с обязательными `author/license/source` (закрывает пробел из аудита — права на медиа).
- Настроить review-workflow в Strapi (публикация только после ревью — без auto-publish).
- Внести 15–20 "якорных" традиций как тестовые данные (placeholder, явно помеченные как черновик, если реальные факты ещё не сверены).

**Files**:
```
/apps/cms/src/api/tradition/content-types/tradition/schema.json
/apps/cms/src/api/tradition-translation/...
/apps/cms/src/api/region/... , /apps/cms/src/api/city/...
/apps/cms/src/api/category/...
```

**Technologies**: Strapi (self-hosted), PostgreSQL (та же БД или отдельная схема — зафиксировать: отдельная схема `cms` в той же PostgreSQL-инсталляции для простоты бэкапа на старте).

**Expected result**: редактор контента может создать традицию на 3 языках через Strapi-админку, увидеть статус верификации, и получить эту традицию через Strapi REST/GraphQL API.

**Testing checklist**:
- [ ] Создание Tradition без хотя бы одного `source_ref` невозможно (обязательное поле в схеме)
- [ ] Попытка опубликовать без прохождения review-шага блокируется
- [ ] API отдаёт `translation_status` и корректно fallback'ится на английский, если перевод для запрошенной локали отсутствует
- [ ] Проверено на 3 реальных (или явно черновых) традициях из разных категорий (§3 спецификации)

**Exit criteria**: минимум 15 традиций в системе со статусом published/needs_verification, API отдаёт мультиязычный контент, редакторский флоу воспроизводим не-разработчиком (проверить: сам фаундер может завести новую традицию, не трогая код).

## Фаза 3 — Публичный контентный сайт

**Goal**: страницы Discover, каталог традиций и страница традиции — рабочий, индексируемый поисковиками сайт без бронирования и без карты (это соответствует Фазе 1 бизнес-roadmap'а — MVP только на контенте).

**Tasks**:
- Настроить Next.js App Router в `apps/web` с i18n-роутингом (`/kk/`, `/ru/`, `/en/` + локализованные slugs, §10.3 спецификации).
- Реализовать главную (Discover) без карты — hero-заглушка, подборки-заглушки на реальных traditions из CMS.
- Реализовать каталог `/traditions` (фильтр по категории, сетка карточек).
- Реализовать страницу традиции `/traditions/[slug]` по 16-блочной структуре §5.1 спецификации и wireframe `tradition-detail` — без блоков "Где научиться" (организаций ещё нет, Фаза 4).
- Настроить SSG/ISR для страниц традиций, `hreflang`, локализованные sitemap.xml, structured data (`FAQPage`, базовый `Article`).

**Files**:
```
/apps/web/app/[locale]/page.tsx                       — Discover
/apps/web/app/[locale]/traditions/page.tsx             — каталог
/apps/web/app/[locale]/traditions/[slug]/page.tsx       — страница традиции
/apps/web/lib/cms-client.ts                             — клиент к Strapi API
/apps/web/app/sitemap.ts, /apps/web/middleware.ts       — i18n роутинг
```

**Technologies**: Next.js (App Router, SSG/ISR), next-intl (или аналог) для i18n, Strapi REST/GraphQL как источник данных.

**Expected result**: реальный, публично доступный сайт с содержательными страницами традиций на 3 языках, проходящий Lighthouse SEO-аудит без критичных ошибок.

**Testing checklist**:
- [ ] Переключение kk/ru/en сохраняет текущую страницу (не сбрасывает на главную)
- [ ] Страница традиции без перевода на выбранный язык корректно показывает fallback, а не пустой блок
- [ ] `hreflang` теги присутствуют и указывают на корректные локализованные URL (проверить через view-source)
- [ ] Core Web Vitals (LCP/CLS) в пределах "Good" на мобильном профиле в Lighthouse
- [ ] Ручная проверка 16 блоков страницы традиции на соответствие wireframe `tradition-detail`

**Exit criteria**: минимум 15 страниц традиций доступны публично на 3 языках, проиндексированы в Google Search Console (сабмит sitemap выполнен), ни одной страницы 500/404 на happy path.

## Фаза 4 — Гео-слой и карта

**Goal**: Region → City → Organization иерархия (§4 спецификации) и рабочая карта — начало соединения "контент → место" без ещё бронирования.

**Tasks**:
- Реализовать в `apps/api` (NestJS) минимальные сущности `Organization` (без booking-полей пока) и связать с `City`/`Region` через PostGIS.
- Реализовать эндпоинт гео-поиска (организации в радиусе/городе).
- Интегрировать Mapbox GL JS на фронтенде — экран `/map` по wireframe `map-desktop`/`map-mobile`, с кластеризацией пинов (§9 дизайн-системы: форма пина зависит от типа).
- Реализовать страницу города `/cities/[region]/[city]` (wireframe `city-profile`) — пока без индекса "культурной активности" (это derived-метрика, отложена).
- Наполнить организациями вручную (Фаза 1 бизнес-roadmap'а — ручной сбор для якорных традиций).

**Files**:
```
/apps/api/src/organizations/{organization.entity.ts,organizations.module.ts}
/apps/api/src/geo/geo.service.ts                        — PostGIS-запросы
/apps/web/app/[locale]/map/page.tsx
/apps/web/app/[locale]/cities/[region]/[city]/page.tsx
/apps/web/components/map/{MapCanvas,ClusterPin,BottomSheet}.tsx
```

**Technologies**: NestJS + TypeORM/Prisma (зафиксировать одно — Prisma, из-за более простой миграционной модели для соло-разработчика), PostGIS, Mapbox GL JS.

**Expected result**: карта показывает реальные организации из БД, клик по кластеру/городу открывает панель со списком (desktop) или bottom sheet (mobile), страница традиции теперь показывает блок "Где научиться" с реальными пинами.

**Testing checklist**:
- [ ] Гео-запрос "организации в городе X" возвращает корректный результат при ручной проверке по нескольким городам
- [ ] Кластеризация визуально работает при зуме карты (пины группируются/разгруппировываются)
- [ ] Bottom sheet на мобильном drag-able, не перекрывает критичный UI карты
- [ ] Форма пина отличается по типу (организация/музей) согласно §9 дизайн-системы

**Exit criteria**: минимум 20 организаций на карте, привязанных к реальным городам, страница традиции связывает контент с картой, мобильная и десктопная версии карты обе рабочие.

## Фаза 5 — Поиск

**Goal**: рабочий мультиязычный поиск (§10.4 спецификации) — "Dombra/Домбра/Домбыра → одна страница".

**Tasks**:
- Развернуть Meilisearch, настроить индексы для Tradition/Organization (мультиязычные поля, синонимы).
- Реализовать курируемые `search_aliases` для якорных традиций вручную (не полагаться на автогенерацию на этом этапе).
- Реализовать собственный слой транслитерации кириллица↔казахская латиница↔английская запись для якорных терминов.
- Реализовать UI автоподсказок (wireframe `search-autocomplete`) и страницу результатов (`search-results`).
- Настроить синхронизацию Strapi → Meilisearch при публикации/обновлении контента (webhook или cron).

**Files**:
```
/apps/api/src/search/{search.service.ts,meilisearch.provider.ts}
/apps/api/src/search/sync/cms-to-index.job.ts
/apps/web/components/search/{SearchInput,Autocomplete,ResultsGrid}.tsx
/apps/web/app/[locale]/search/page.tsx
```

**Technologies**: Meilisearch, cron/webhook-синхронизация (BullMQ + Redis для очереди синка).

**Expected result**: запрос "Dombra" на английском интерфейсе и "Домбыра" на казахском находят одну и ту же страницу традиции; автоподсказки группируются по типу результата.

**Testing checklist**:
- [ ] Ручной тест 10 известных синонимов/альтернативных транслитераций для якорных традиций — все находят правильную страницу
- [ ] Опечатка в запросе (fuzzy match) всё ещё находит релевантный результат
- [ ] Обновление традиции в Strapi отражается в поиске без ручной переиндексации (проверить задержку синка)
- [ ] Автоподсказки не блокируют ввод (debounce настроен, нет визуального дребезга)

**Exit criteria**: поиск работает по всем 3 языкам для всех якорных традиций, синхронизация CMS→индекс автоматическая, задержка синка задокументирована (и приемлема для контента, который не меняется поминутно).

**Goal**: пользователи могут войти, иметь профиль, привязать детские профили — до старта бронирования (booking требует авторизованного пользователя).

- Реализовать модалку логина (`modal-login`), вызываемую по требованию авторизации с любого экрана.

**Files**:

**Technologies**: NestJS Passport (OAuth-стратегии), SMS-провайдер для OTP (локальный, поддерживающий номера РК), JWT/сессии.

**Expected result**: пользователь может зарегистрироваться тремя способами, увидеть и отредактировать профиль, создать детский профиль с явным чекбоксом согласия на будущую публикацию медиа.

**Testing checklist**:

- [ ] Выход и повторный вход сохраняет профиль и настройки языка/темы

**Exit criteria**: полный флоу онбординга (wireframe `onboarding`) работает end-to-end, минимум один реальный тестовый аккаунт с детским профилем создан и проверен.
