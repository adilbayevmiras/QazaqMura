## 1. Продуктовое видение

**One-liner**: Qazaq Mūra отвечает на два вопроса подряд — «Что это за традиция?» и «Где я могу это увидеть/потрогать/научиться?» — за один визит, на трёх языках, с картой и записью в один клик.

**Проблема рынка**: культурный контент о Казахстане разбросан — Wikipedia даёт текст без действия, Instagram — картинку без контекста, турагентства — тур без глубины, гос. порталы — факты без UX. Никто не соединяет «знание» и «действие» в одном продукте.

```
Wikipedia (глубина знания) + Airbnb Experiences (бронирование) + Google Maps (гео)
+ ClassPass (регулярные занятия) + GetYourGuide (билеты/экскурсии) = Qazaq Mūra
```

| Аудитория | Что получает |
|---|---|
| Турист | Маршрут "что аутентичное посмотреть" + билеты |
| Житель Казахстана | "Забытые" ремёсла рядом с домом, кружки для детей |
| Родитель | Секцию для ребёнка за 3 клика, с ценой и расписанием |
| Школьник/студент | Структурированный источник с проверенными фактами |
| Иностранец/диаспора | Смысловой слой "почему это важно" |
| Исследователь культуры | Кросс-referenced каталог с источниками ЮНЕСКО/музея |

## 4. Географическая модель (Region → City → Location)

Ранее традиция ссылалась на `region_ids`, а организация — на `city_id` без общей иерархии. Зафиксировано единое дерево:

```yaml
Region:         # 20 областей/городов респ. значения РК
  id, slugs {kk,ru,en}, translations[]

City:
  id, region_id, slugs {kk,ru,en}, lat, lng, translations[]

# Всё остальное (Organization, Master, Event) ссылается на city_id,
# и получает region_id вычисляемо через City → Region.
# Tradition.region_ids остаётся — обозначает происхождение традиции,
# а не место, где её сейчас можно изучить (это разные вещи).
```

Это разграничивает «где традиция зародилась» (справочно, на странице традиции) от «где её сейчас можно освоить» (операционно, через City у Organization/Master/Event) — путаницы между ними больше нет.

## 5. Шаблон страницы традиции и модель данных

### 5.1 Структура страницы традиции
Hero → краткое описание → info-strip (регион происхождения, heritage-статус, сезонность, возраст) → история → значение → почему важно сохранить → интересные факты → фотогалерея → видео → карта "где учиться" → организации/мастера → агрегированные цена/возраст/расписание → отзывы → FAQ → похожие традиции → источники и дата сверки.

### 5.2 Data model (заморожено для v1.0)

```yaml
Tradition:
  id, slugs {kk,ru,en}
  translations: [{locale, title, short_description, history, meaning,
                   why_preserve, faq[], translation_status: enum [human, machine_draft]}]
  category_ids: [], region_ids: []          # регион происхождения, справочно
  heritage_status: enum [unesco, national_list, regional, undocumented]
  media: {hero, gallery: [{url, author, license, source}], videos[]}   # права на медиа — обязательное поле
  interesting_facts: [{locale, text}]
  related_tradition_ids: []
  source_refs: [{source, url, retrieved_at}]
  seo: {meta_title, meta_description}       # per locale
  status: enum [draft, needs_verification, published]
  last_verified_at

Organization:
  id, name, type: enum [school, club, museum, ethno_village, festival_org]
  tradition_ids: [], city_id
  pricing: {currency: "KZT", from, packages[]}     # KZT — базовая валюта расчётов, см. §9
  age_range, schedule[]
  media, rating, review_ids[]
  verification_status, premium_tier
  team_members: [{user_id, role: enum [owner, staff]}]   # роли внутри аккаунта организации

Master:
  id, name, photo, bio (per locale)
  tradition_ids: [], organization_id (nullable)
  city_id
  verification: {status: enum [pending, verified, official], verified_by, verified_at, evidence_refs[]}
  badge: enum [none, verified_by_qm, official_master]
  payout_profile: {tax_status: enum [self_employed_patent, individual_entrepreneur, agent_scheme],
                    payout_method, verified_at}     # см. §9.2
  experience_ids: [], rating, review_ids[]

Experience (bookable unit):
  id, organization_id (nullable), master_id (nullable)   # хотя бы одно из двух обязательно
  tradition_id
  type: enum [class, tour, ticket, workshop, camp, family_event]
  price, capacity, calendar/slots, age_range, is_kid_friendly
  cancellation_policy, payout_hold_hours: int      # см. §8.4 (диспуты)

Review:
  id, author_id, booking_id            # ОБЯЗАТЕЛЬНО — отзыв только по завершённой брони
  target: {type: enum [tradition, organization, master, experience], id}
  rating, text (locale), media[]
  moderation_status: enum [pending, approved, rejected]

CommunityPost:
  id, author_id, type: enum [photo, article, route]
  linked_entity: {type: enum [tradition, organization, event, master], id}
  content, media[], locale
  moderation_status: enum [pending, approved, rejected]
  contains_minors: boolean              # включает доп. модерацию, см. §13.2
  reputation_weight
```

**Ключевое разделение**: Tradition = контент, Organization/Master = поставщики, Experience = транзакция, Review = только по факту покупки, CommunityPost = свободный UGC с отдельной модерацией.

## 6. Карта Казахстана

Кластеризованные пины по стране → клик по городу → зум + панель (организации, клубы, музеи, школы, мастера, фестивали) с фильтрами по традиции/цене/возрасту/дате. Слой "сейчас" (события за 7 дней), переключение слоёв ("где учат"/"где увидеть"/"фестивали"/"исторические места"), режим "Маршрут". Мобильная версия — bottom sheet.

**Технически**: Mapbox GL JS + собственный geocoding-слой для казахских топонимов на 3 языках, гео-запросы идут через PostGIS (см. §12), а не дублируются в поисковом движке — единый источник истины для координат.

## 7. Продуктовые модули

### 7.1 AI-гид (`/guide`)
Не чат-бот, а route-builder поверх реального инвентаря — компонует существующие Tradition/Organization/Master/Experience/Event, а не выдумывает их. LLM используется только как NLU-слой и генератор пояснений, никогда — как источник фактов о местах/ценах/расписании.

**Флоу**: свободный текст или фильтры → парсинг intent → **обязательный шаг подтверждения** ("Правильно понял: Алматы, 2 дня, с детьми?") → запрос к API поиска/бронирования → черновой маршрут по слотам с реально доступными Experience/Event → редактирование пользователем → бронирование → опциональная публикация в `/community/routes`.

MVP без LLM (route-builder на фиксированных фильтрах) — Фаза 2. Полноценный диалоговый слой — Фаза 4 (см. §16).

### 7.2 Раздел для детей (`/kids`)
Отдельная точка входа (не фильтр внутри общего каталога) — у родителя другой search intent. Лагеря / кружки / летние школы / семейные мероприятия. Фильтры: возраст, формат (разовое/регулярное/с проживанием), нужен ли сопровождающий. Родительский профиль хранит несколько детских профилей. Отдельные вопросы в форме отзыва (безопасность, вовлечённость). Юридические требования к детским данным — см. §13.2.

### 7.3 Верифицированные мастера (`/masters`)
Третий, персональный уровень доверия — поверх heritage-статуса традиции и рейтинга организации.

| Бейдж | Критерий | Кто присваивает |
|---|---|---|
| 🏛 Verified by Qazaq Mūra | Подтверждена личность + реальная деятельность | Модерация платформы |
| 🏅 Official Master | Признание госорганизации/ассоциации/музея | Внешняя институция + платформа |

Флоу: заявка (`/for-masters`) → портфолио/документы → модерация → профиль с историей и "у кого учился". Выплаты мастерам-фрилансерам — см. §9.2 (это блокирующий вопрос, не деталь UX).

### 7.4 Культурный календарь (`/calendar`)
Фестивали, айтысы, турниры, выставки, обрядовые праздники, дни открытых дверей музеев. Виды "Календарь"/"Лента", фильтры по региону/категории, подписки с уведомлениями (сервис уведомлений — см. §12). Каждое событие связано с `tradition_ids`; AI-гид использует календарь как источник дат.

### 7.5 Сообщество (`/community`)
Фото/видео, статьи, маршруты — все через модерацию (auto-flag + human review), визуально отделены от редакционного контента (§14). Репутационная система авторов. Отдельная пометка контента с участием несовершеннолетних (`contains_minors`) — усиленная модерация, см. §13.2.

### 7.6 Культурный паспорт (`/passport`)
Чек-лист "посещено/изучено" по категориям с прогресс-баром, автоматическая отметка при завершённой брони vs ручная "я уже знаю это" (визуально отличается). Достижения по паттернам поведения. Публичный/приватный режим. Сертификаты от мастеров/организаций подгружаются как подтверждённые уровни. Точка пересечения всех модулей: брони заполняют паспорт, мастера выдают сертификаты, AI-гид использует "непосещённое" как параметр.

## 8. Доверие, отзывы и модерация

### 8.1 Иерархия trust-бейджей (зафиксировано, чтобы избежать визуального шума)
Порядок отображения на карточке (сверху вниз по значимости, не более 2 одновременно видимых, остальное — в тултипе/на странице):
1. Heritage-статус традиции (уровень контента, не поставщика) — показывается только на странице традиции, не на карточке организации.
2. Бейдж мастера (Official > Verified) — если Experience ведёт мастер.
3. Бейдж организации (Premium/Verified) — если ведёт организация без привязанного мастера.

### 8.2 Правило отзывов
Отзыв возможен только с `booking_id` завершённой брони (см. модель `Review`, §5.2). Ручных отзывов "просто так" нет — это устраняет основной вектор накрутки.

### 8.3 Модерация UGC
Весь пользовательский контент — через очередь модерации (auto-flagging по ключевым словам + human review), без auto-publish. Контент с несовершеннолетними — отдельная, более строгая очередь (§13.2).

### 8.4 Диспуты и возвраты
`Experience.payout_hold_hours` — средства за бронирование удерживаются платформой N часов после завершения Experience перед выплатой поставщику (эскроу-логика), что даёт окно для обращения при споре "занятие не провели" / "клиент не пришёл". Стандартные правила возврата фиксируются в единой политике (`/legal/refund-policy`), локальные исключения по каждой Experience — через `cancellation_policy`. Споры выше стандартного порога — разбирает саппорт-команда платформы вручную (не автоматическое решение в пользу одной из сторон).

## 9. Монетизация

| Модель | Механика | Кому |
|---|---|---|
| Premium-размещение | Топ в поиске/карте, аналитика, бейдж | Организации, мастера |
| Комиссия за бронь | 10–15% на старте (growth-стратегия) | Все bookable Experience |
| Реклама мероприятий | Продвижение в подборках/на карте | Организаторы, акиматы |
| Продажа билетов | Собственный чекаут | Музеи, этноаулы |
| Партнёрки | Affiliate с отелями/авиакомпаниями | B2B travel |
| Донаты | "Adopt a tradition", прозрачный отчёт | Частные лица, диаспора |
| B2B/B2G лицензирование данных | API/виджет верифицированного контента | Партнёры |
| Freemium School Mode | Учебные модули, сертификаты по подписке | Школы, родители |

### 9.1 Валюта
Базовая валюта расчётов — **KZT** (`Organization.pricing.currency`). Для иностранных туристов — клиентское отображение конвертированной суммы по курсу на момент просмотра (информативно), фактическое списание — через Stripe в валюте карты либо в KZT через локальный эквайринг; курсовая разница не является предметом спора платформы.

### 9.2 Выплаты независимым мастерам (закрывает P0/P1-риск из аудита)
Комиссионная модель не работает без легального механизма выплат физлицам без юрлица. Решение для v1.0:
- Мастер при регистрации указывает `payout_profile.tax_status`: самозанятый по патенту / ИП / согласие на агентскую схему (платформа выступает налоговым агентом по договору возмездного оказания услуг — стандартная практика для маркетплейсов услуг физлиц в РК).
- Выплата без легального статуса **не производится** — это верификационный гейт, а не опция; отражено в `payout_profile.verified_at`.
- Технически — отдельный Payouts-сервис (см. §12), не часть общего Payments-флоу с организациями (у организаций — прямые договоры и Kaspi Business/эквайринг, у мастеров-физлиц — агентская схема с удержанием налога платформой).

## 10. Мультиязычность

### 10.1 Архитектура
Обязательные языки: kk, ru, en. Расширяемо (tr, de, fr, ar, zh, es) без переписывания схемы — locale как измерение, не колонка. Fallback на английский при отсутствии перевода. RTL-готовность заранее (CSS logical properties).

### 10.2 Политика перевода по уровням (закрывает риск стоимости при масштабе)
| Уровень контента | Требование |
|---|---|
| Страницы традиций (редакционный контент) | Только человеческий перевод, ревью носителем/экспертом обязательно |
| Карточки организаций/мастеров, описания Experience | Черновой машинный перевод по умолчанию с видимой пометкой "предварительный перевод"; человеческая проверка — часть Premium-тарифа организации |
| UGC (отзывы, статьи сообщества) | Не переводится централизованно; отображается в оригинале + опциональный machine-перевод по запросу пользователя (как в Google Maps reviews) |

`translation_status` в data model (§5.2) фиксирует это поле явно.

### 10.3 URL и SEO
Локализованные slugs (`qazaqmura.kz/kk/dastur/dombyra`), `hreflang` на все локали + `x-default`, локализованные sitemap.xml, structured data (schema.org) на каждом языке. Раздельная SEO-стратегия под казахоязычный/русскоязычный/англоязычный интент-кластеры, Search Console + Bing Webmaster по локалям, SSR/SSG для Core Web Vitals.

### 10.4 Поиск
Задача: `Dombra` / `Домбра` / `Домбыра` → одна страница. Курируемые `search_aliases` для якорных традиций + собственный transliteration-мэппинг (кириллица ↔ новый казахский латинский алфавит 2021+) + fuzzy-matching. Embedding-based semantic search — второй слой, запланирован на Фазу 3 (не на старте, см. §16), чтобы не утяжелять MVP инфраструктурой векторного поиска раньше необходимости.

## 11. UX / Дизайн-система

**Референсы поведения**: Apple (типографика), Airbnb (карточки, доверие через фото/отзывы), Notion (лонгриды), Stripe (формы/чекаут), Google Maps (гео-интеракция).

**Принципы**: контент прежде хрома; воспринимаемая скорость (skeleton-состояния, оптимистичный UI); glassmorphism только на плавающих панелях поверх фото/карты, никогда на тексте; тёмная тема — отдельная палитра, не инверсия; motion только со смыслом, без decorative-only анимаций, замедляющих LCP.

**Ключевые экраны**: Discover, страница традиции, карта (fullscreen + bottom sheet), профиль организации/мастера, флоу бронирования (3 шага, паттерн Stripe Checkout), паспорт, поиск, dashboard организации (с учётом ролей `team_members`, §5.2).

**Токены**: variable font с поддержкой кириллицы и казахской латиницы; радиусы 8/12/20px; мягкие многослойные тени (light) / glow-based (dark); grid 12 колонок, брейкпоинты 375/768/1024/1440/1920; единая icon-library (line-style).

## 12. Техническая архитектура (зафиксированные решения)

```
Frontend:     Next.js (App Router, SSG/ISR для страниц традиций)
              + Mapbox GL JS + i18n-роутинг через middleware

Backend:      NestJS (Node.js/TypeScript) — единый язык с фронтендом,
              упрощает найм и переиспользование типов между слоями.
              Отдельные сервисы: Content, Search, Booking, Payments,
              Payouts (см. §9.2), Reviews, Notifications, Analytics.

Database:     PostgreSQL + PostGIS (гео-запросы — единый источник истины,
              карта и фильтры по местоположению идут через него, не дублируются в поиске)
              + Redis (кэш, сессии, rate-limiting)

Search:       Meilisearch — выбран за простоту эксплуатации и typo-tolerance
              "из коробки" для мультиязычного fuzzy-search на масштабе стартапа;
              embedding-слой добавляется в Фазе 3 (см. §10.4)

CMS:          Strapi (self-hosted, открытый код, контроль над данными
              и полем translation_status/last_verified_at без вендор-лока)

Media:        Cloudflare Images/Stream — оптимизация под 3G/4G в регионах РК

Payments:     Kaspi Business/локальный эквайринг (организации, прямой договор)
              + Stripe (международные карты) + отдельный Payouts-сервис
              с агентской налоговой схемой для мастеров-физлиц (§9.2)

Notifications: отдельный сервис (email + push), очередь на события календаря/бронирований

Analytics:    событийный трекинг (напр. PostHog self-hosted) →
              витрина данных для дашборда организаций ("аналитика просмотров", §9)

Infra:        Frontend — Cloudflare edge. Backend/DB — регион с учётом
              требований локализации персональных данных РК (см. §13.1);
              по умолчанию — локальный/региональный дата-центр для PII,
              non-PII медиа — глобальный CDN.

Auth:         OAuth (Google/Apple) + локальные номера телефонов
```

## 13. Юридический и комплаенс-слой

### 13.1 Локализация данных
Закон РК «О персональных данных и их защите» требует учитывать трансграничную передачу и, в ряде случаев, хранение данных резидентов на территории РК. **Решение по умолчанию для v1.0**: персональные данные пользователей (профиль, брони, платёжные токены) хранятся в региональном/локальном дата-центре; неперсонализированные медиа и контент традиций — на глобальном CDN. Финальное юридическое заключение — обязательный шаг Фазы 0, до продакшн-запуска.

### 13.2 Данные несовершеннолетних
- Детский профиль создаётся и управляется только через аккаунт родителя/опекуна — не самостоятельно.
- Публикация фото/видео с детьми в `/community` и на карточках Experience — только с явным согласием родителя (чекбокс при загрузке, отдельный от общего согласия на публикацию).
- `CommunityPost.contains_minors = true` → усиленная модерация (человек, не только auto-flag), без исключений.
- В отзывах на детские Experience (§7.2) запрещено указывать полное имя ребёнка — только имя/инициал по умолчанию в форме.

### 13.3 Диспуты, возвраты, выплаты
См. §8.4 (эскроу-логика через `payout_hold_hours`) и §9.2 (легальный статус выплат мастерам) — оба пункта являются юридической, а не только продуктовой зависимостью и должны быть согласованы с юристом до запуска монетизации (Фаза 2).

## 14. Политика источников и достоверности контента

1. Каждая страница традиции — источник и дата последней сверки.
2. Бейдж достоверности: ✅ UNESCO-recognized / 📖 National List / 🏘 Community-documented.
3. Недостаточно данных → явная пометка "информация уточняется", не предположение.
4. CMS с обязательным review-шагом перед публикацией, без auto-publish AI-текста.
5. Открытая страница `/sources` со ссылками на gov.kz / unesco.org / nmrk.kz.

## 15. Метрики успеха

**North Star**: количество завершённых "путешествий понимания→действия" (просмотр традиции → клик "найти организацию/мастера" → бронирование/визит).

**Supporting**: конверсия просмотр→бронь, retention (возврат за новой традицией), NPS партнёров (организации и мастера отдельно), доля органического трафика по языкам, время до первой брони, доля Experience с payout-диспутами (индикатор здоровья маркетплейса, целевое значение <2%).

## 16. Единый Roadmap

**Фаза 0 (до кода)**: юридическое заключение по §13.1/§13.2/§9.2; сверка контента с gov.kz/UNESCO/nmrk.kz для 15–20 якорных традиций.

**Фаза 1 (MVP, 8–12 недель)**: страницы традиций (3 языка, без бронирования) · карта с вручную собранными организациями · поиск с ручными алиасами для якорных традиций · контакт с организацией напрямую (email/телефон), без платежей.

**Фаза 2**: бронирование + Payments/Payouts (включая агентскую схему для мастеров) + Reviews (с привязкой к `booking_id`) + B2B dashboard · `/kids` как точка входа · базовый route-builder без LLM.

**Фаза 3**: `/masters` с верификацией и бейджами · `/calendar` как полноценный слой · UGC MVP (фото/отзывы, без статей/маршрутов) · культурный паспорт и геймификация · мобильное приложение · embedding-based semantic search.

**Фаза 4**: полноценный диалоговый AI-гид (LLM-слой поверх route-builder) · статьи/маршруты в community · AR/VR.
